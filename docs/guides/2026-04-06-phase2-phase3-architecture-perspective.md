# Phase 2 & Phase 3 架构视角总结

> 面向：**架构决策与取舍**，少代码、多「为什么」。  
> 依据：`2026-03-30-markdown-rendering-design.md`、`2026-03-31-conversation-persistence-design.md` 与当前实现方向。  
> 日期：2026-04-06

---

## 1. 系统语境（这两阶段在整体里干什么）

English Tutor 是 **浏览器中的对话式英语辅导**：用户发文本，后端调用 LLM（及分类、RAG、工具等）生成结构化文本回复。  

- **Phase 2** 解决的是 **表达层**：回复仍是字符串，但**呈现**从纯文本升级为可读性更好的富文本。  
- **Phase 3** 解决的是 **状态层**：对话不再绑定单次页面生命周期或单次进程内存，而是**可恢复、可清空**的连续记忆。

二者正交：一个管「怎么显示」，一个管「存哪、谁为真」。

## 2. Phase 2：Markdown 渲染 — 方案与取舍

### 2.1 要解决的问题

LLM 习惯用轻量 Markdown（粗体、列表、分段）组织答案；纯文本展示会损失层次，降低阅读体验。

### 2.2 采用的技术方案

- `**react-markdown`**：在 React 树中把 Markdown 解析为**元素节点**，而不是先变成一大段 HTML 字符串。  
- `**@tailwindcss/typography`（prose）**：为解析出的语义标签提供**一致、可维护**的排版，避免手写一堆针对 `p`/`ul`/`strong` 的零碎 CSS。

### 2.3 为什么这样选

1. **安全默认**：不走「字符串 HTML + 注入 DOM」的典型路径，降低 **XSS** 面；英语辅导场景仍要面对不可信模型输出，防御成本要低。
2. **与 React 一致**：后续若做流式输出、按块替换组件（如 TTS、自定义块），**组件化**扩展比纯 HTML 串更顺。
3. **范围可控**：当前场景不需要表格、任务列表、代码高亮；**少依赖、少分支**符合 YAGNI，也减小包体与维护面。

### 2.4 明确不用的路线及原因


| 路线                                         | 为何不采用                                             |
| ------------------------------------------ | ------------------------------------------------- |
| `**marked` 等 + `dangerouslySetInnerHTML`** | 需要额外引入净化与策略维护；失去 React 组合能力；安全与演进成本偏高。            |
| **GFM + 语法高亮全家桶**                          | 当前 LLM 输出形态用不到；依赖与构建复杂度上升，违反「只解决当前问题」。            |
| **用户消息同样 Markdown**                        | 用户输入不可当作「可信排版」；误解析会带来困惑与安全顾虑；产品上要区分「我打的字」和「模型排版」。 |


### 2.5 架构上留下的钩子

同一套「模型产出字符串 → 前端渲染」管道，未来可接 **流式增量 `content`** 或 `**ReactMarkdown` 的自定义组件映射**（语音、富媒体），而不必推翻存储与 API 形态。

## 3. Phase 3：对话持久化 — 方案与取舍

### 3.1 要解决的问题

- **刷新/重进页面**：UI 应能恢复**用户可见**的近期对话。  
- **进程重启**：LLM 侧**上下文**（摘要 + 近期轮次）应能恢复，避免「从头陌生人」。  
- **产品配套**：用户需要**主动清空**记忆的入口（重新开始）。

### 3.2 核心架构判断：两套「真相」

设计上把问题拆成两件事：

1. **给模型用的上下文**：要控制 **token**，需要 **摘要 + 裁剪后的多轮 history**（随压缩策略变化）。
2. **给人看的展示历史**：需要 **稳定、按时间可追溯的原文**，且条数有上限（如最近 30 条）。

若只用一份结构同时服务两者，**压缩 history 会导致界面上消息「消失」**，体验与心智模型冲突。  

因此架构上采用 **双轨存储**：  

- 一条轨存 **LLM 会话状态**（摘要 + history JSON）；  
- 另一条轨存 **展示消息流水**（逐条追加、按时间截取）。

这是 Phase 3 的**第一性原理**，比具体用哪种数据库更重要。

### 3.3 存储与中间层技术选择


| 决策      | 方案                           | 为什么                                                              |
| ------- | ---------------------------- | ---------------------------------------------------------------- |
| 单机持久化引擎 | **SQLite**（`better-sqlite3`） | 零独立数据库服务、单文件易备份与理解、SQL 可迁移；与「单用户、单部署单元」阶段匹配。                     |
| 同步驱动    | **同步 API + 事务**              | 编排逻辑简单，路由层用事务包裹「写会话 + 写两条消息」，**原子性**清晰。                          |
| 与业务解耦   | **Repository 层**             | 把 SQL 收敛在少数模块；未来换 **PostgreSQL** 或云托管库时，**控制爆炸半径**（spec 明示演进路径）。 |


### 3.4 为何现阶段不默认 PostgreSQL

- **需求未触发**：无多租户、无多实例强一致写入、无托管运维刚需时，PostgreSQL 带来**运维与成本**，收益不足。  
- **迁移路径已预留**：数据访问集中在 repository，**不是**「业务 SQL 散落各处」的形态；等产品化（多用户、水平扩展、云 RDS）再换存储，是**有意识的延后**，而非技术债失控。

### 3.5 会话模型：后端固定「唯一会话」

- **不在前端传 `sessionId`**：单用户产品下，避免客户端状态与「谁是谁的会话」纠缠；跨设备只要打同一后端，即同一记忆。  
- **代价（架构上承认的）**：所有连到**同一后端实例**的客户端**共享**这一段记忆；与「私教一对一、非协作工具」的定位一致。多用户阶段需引入 **身份 + 每用户多条 session** 或等价模型。

### 3.6 一致性与失败策略（架构语义）

- **仅在对话成功完成后持久化**：失败时不写库，DB 保持上一次一致快照；允许短时内存与磁盘不一致，由**下一次成功写**收敛。  
- **同一事务写「会话状态 + 两条消息」**：避免「模型状态已前进、展示缺一条」的裂口；**重置**同样走事务，清空展示轨与会话轨。

### 3.7 明确不用的路线及原因


| 路线                      | 为何不采用（本阶段）                    |
| ----------------------- | ----------------------------- |
| **前端 localStorage 为权威** | 与「后端掌握上下文、多终端一致」冲突；密钥与策略也难统一。 |
| **只用 LLM history 做列表**  | 与压缩策略冲突，见 §3.2。               |
| **继续维护 CLI 为平行入口**      | 聚焦 Web 交付，减少入口与状态同步面。         |

## 4. 两阶段如何拼成一条产品架构故事

- **Phase 2**：在**不改变后端契约**的前提下，提升**人类可读性**（表现层）。  
- **Phase 3**：在**不改变 LLM 核心服务职责**的前提下，把**状态外置**到可恢复存储，并用 API 定义**加载历史**与**重置**（状态层）。

合起来：用户看到的是 **持久化后的消息列表 + Markdown 渲染的助手回复**；模型吃的是 **独立维护的、可压缩的上下文**。

## 5. 演进路线与产品化升级路径

以下内容摘自 `[2026-03-31-conversation-persistence-design.md](../superpowers/specs/2026-03-31-conversation-persistence-design.md)` **§9 / §9.1**，在架构总结中集中呈现，便于与本文 §3 对照。

### 5.1 演进路线（产品阶段树）

```
Phase 1（MVP 文字聊天）✅ 已完成
  ├── Phase 2（Markdown 渲染）✅ 已完成
  ├── Phase 3（对话记忆持久化）← 本次实现
  ├── 后端：LangChain 改造
  └── Phase 4（Streaming 响应）
        └── Phase 5（TTS 语音）
              └── Phase 6（数字人）
```

**架构上如何读这棵树：**

- **Phase 2 / 3** 与后续 **Phase 4～6** 是同一产品轴上的纵向深化：先可读、可记，再实时流式、多模态。  
- **「后端：LangChain 改造」** 与持久化正交：动的是编排/抽象层，不否定 Phase 3 的「双轨存储 + Repository」结论。  
- **Phase 4（Streaming）** 落地时，需在实现层单独设计：**增量内容**与 **落库/展示** 的竞态、部分失败与重试语义（本文 §2.5 的表现层钩子仍成立）。

### 5.2 产品化升级路径（存储与多租户）

当项目从个人工具走向**产品化**时，spec 中规划的存储演进如下：

```
Phase 3（当前）: SQLite 单文件，单用户
    ↓ 产品化
├── PostgreSQL 替代 SQLite（多用户、高并发、云托管）
│   └── 改动范围：仅 src/db/ 内的 3 个文件
│       - database.ts: better-sqlite3 → pg + 连接池
│       - session-repo.ts: 同步 → async，SQL 语法微调
│       - message-repo.ts: 同步 → async，SQL 语法微调
│
├── Redis 缓存层（可选，叠加在 PostgreSQL 之上）
│   └── 热 session 缓存：getDefaultSession() 先查 Redis，miss 再查 PG
│   └── 适用于高频访问场景，单用户阶段不需要
│
└── 用户认证 + 多租户
    └── sessions 表 + messages 表加 user_id 列
    └── session-manager 从单例改为按 user_id 查找
```

**与架构决策的挂钩（spec 原文意图）：**

- **Repository 模式** 的目标是：更换存储引擎时，**上层业务**（如 `chat-service`、路由编排）尽量**不因 SQL/驱动细节而改动**。  
- 若将来把 repo 接口统一为 `**async`**，迁移到 PostgreSQL 时有机会 **只改 repository 实现**，调用方变更面更小（spec 备忘）。

**与本文 §3.4 / §3.5 的关系：** 当前 **SQLite + 后端唯一会话** 对应上图「Phase 3（当前）」；**PostgreSQL / Redis / user_id** 对应「产品化」分叉，**非**当前阶段的默认承诺。

### 5.3 补充：除上图已列能力外的实现提醒

- **多应用服务器 + 负载均衡** 时，磁盘上的单文件 SQLite **无法自然多机共享**；产品化路径中 **PostgreSQL（或等价共享存储）** 与 **会话亲和性或无状态 + 中央库** 是常规解法，与 §5.2 一致。
